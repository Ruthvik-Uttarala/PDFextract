PDFextract  Product Requirements Document
## 1. Document Purpose
This document defines exactly what the PDFextract product is, who it is for, what problems
it solves, what functionality is included in the MVP, what is explicitly out of scope, and
what “done” means for the first production-ready demo release.
This PRD is the first canonical document for the project. It is intentionally specific so
Codex, engineers, designers, and future contributors do not guess requirements, invent
features, or change architecture without approval. This document should be treated as the
source of truth for product behavior. App flow, backend structure, technical
implementation, and design guidelines must all follow this document.
This PRD supersedes earlier exploratory notes that mentioned alternate stacks such as
Firestore, Framer, Vercel-only backend patterns, or generic AI-agent architectures. Those
earlier notes remain useful for understanding the original problem framing and use cases,
but this PRD locks the product definition to the meeting-approved stack and delivery
model. The original notes still matter because they established the core extraction goals,
the multi-step AI processing concept, the invoice and research-document use cases, and
the MVP expectation of downloadable structured output.

## 2. Product Name
Product Name: PDFextract
Working Tagline: AI-powered PDF extraction for structured business output
Internal Positioning: A client-facing AI document processing system that turns
unstructured PDF files into validated, structured, downloadable Excel outputs.


## 3. Product Vision
PDFextract is a web-based AI document processing platform that allows authenticated
users to upload PDF files, trigger automated extraction pipelines, and download structured
Excel outputs produced from the content of those documents.
The first version of the product is not a generic “chat with PDF” experience. It is a focused
operational workflow product. The user’s job is simple: upload a document, track
processing status, review results, and download the processed output. The system’s job is
to handle ingestion, storage, queuing, extraction, validation, persistence, and delivery.
The longer-term vision is to make PDFextract a reusable document automation platform
that can handle invoices, contracts, reports, and research documents using configurable
schemas and multiple extraction pipelines. But the MVP is deliberately narrower: it must
prove reliable end-to-end extraction for the first supported document classes and produce
outputs that users can actually use in business workflows.

## 4. Problem Statement
Organizations receive large volumes of PDFs that contain valuable data but are difficult
and expensive to process manually. These documents may include invoices, statements,
scientific reports, clinical notes, contracts, and other semi-structured or unstructured
content. Teams often open each file manually, locate key fields by hand, enter them into
accounting systems or spreadsheets, and then archive both the source and output files.
This process is slow, repetitive, error-prone, and not scalable.
The core opportunity behind PDFextract is to remove manual extraction work from the
user’s workflow. Instead of reading each PDF and typing data into spreadsheets or
systems, the user uploads the file once and receives a structured output generated by AI.
Earlier project notes framed this clearly: the target problem is manual PDF-to-structured-
data work, especially for invoices and research documents, with an MVP centered on
upload, extraction, validation, and generated output rather than deep downstream ERP
integration.


## 5. Target Users
## Primary Users
The primary users for the MVP are operational and business users who need structured
data from PDFs but do not want to manage technical extraction logic themselves.
Examples include:
- Accounting and finance teams processing invoices
- Operations teams handling vendor or compliance documents
- Research or clinical operations teams extracting data from scientific PDFs
- Internal analysts who need document content in spreadsheet form
These users are not expected to understand Kafka, Lambda, Gemini prompts, or schema
mapping internals. They care about accuracy, visibility, speed, and exportability.
## Secondary Users
Secondary users include:
- Internal admins overseeing processing health
- Client stakeholders evaluating demo readiness
- Developers maintaining extraction schemas and pipelines
- Business owners reviewing extraction performance and failure modes
Admin/Operator Personas
A limited admin/operator layer is required for the MVP so internal teams can:
- view uploaded files and job status
- inspect failures
- re-run jobs when needed
- review extraction outputs
- monitor processing pipeline health at a basic level


## 6. Core Use Cases
The MVP is built around two anchored use cases derived from the project notes.
## Use Case A: Invoice Extraction
A user uploads an invoice PDF. The system stores the source file, triggers processing,
extracts text and table content, identifies invoice-specific fields, validates the result,
generates an Excel output, stores the processed artifact, and makes it downloadable from
the web interface.
Typical extracted fields may include:
- invoice number
- invoice date
- vendor name
- billing address
- shipping address if present
- purchase order or contract reference
- subtotal
- tax
- total amount
- currency
- line items with description, quantity, unit price, and line total
- payment terms
- due date
## Use Case B: Research / Scientific Document Extraction
A user uploads a scientific or clinical PDF. The system processes the file, extracts key
metadata and structured content, and generates a standardized export suitable for
downstream analysis or knowledge-base ingestion.
Typical extracted fields may include:
- document title
- author or institution
- publication or submission date

- subject or study type
- summary or abstract
- key entities
- tabular measurements if present
- section-based extracted data according to configured schema
Use Case C: Status Tracking and Download
A user uploads one or more PDFs and needs to know:
- whether the files were accepted
- whether they are queued, processing, completed, or failed
- when the output is ready
- where to download the resulting Excel file
## Use Case D: Internal Retry / Failure Review
An internal operator reviews failed jobs, sees the failure reason, and retries the processing
flow after configuration correction or transient issue resolution.

## 7. Product Goals
The MVP goals are:
Goal 1: End-to-End Functional Pipeline
The product must support a complete workflow from authenticated upload to
downloadable processed output with no manual engineering intervention required for a
normal successful run.
## Goal 2: Structured Output Users Can Actually Use
The output must be a real Excel file stored in S3 and surfaced in the app, not merely raw AI
text. The result should be usable for accounting, reporting, or analyst workflows.

## Goal 3: Clear Operational Visibility
Users must be able to see the status of each upload. The system should not feel like a
black box.
Goal 4: Modular AI Extraction
The product must support document-type-aware extraction so the system can evolve from
invoice-first toward broader document categories over time.
## Goal 5: Strong Demo Readiness
The first release must be credible in front of a prospective client. That means the workflow
should feel deliberate, polished, and operationally coherent, not like a toy prototype.

- Non-Goals
The following are explicitly out of scope for the MVP:
- Full ERP or SAP integration
- Human-in-the-loop editing of extracted fields inside the web app
- Multi-tenant enterprise administration with granular RBAC
- Real-time collaborative review
- OCR support for every possible scan quality edge case
- Arbitrary schema builder for non-technical users
- Billing and subscription management
- Public API for external clients in v1
- Full audit/compliance suite
- Bulk batch uploads at massive enterprise scale
- Complex analytics dashboards
- Fine-tuned custom model training
- Chatbot-style conversational document QA
- Native DOCX/image-first workflows as first-class upload types
- PDF annotation UI
- Versioned workflow orchestration UI
- On-prem deployment

These can be considered later, but they must not leak into the MVP build unless explicitly
added through a revised PRD.

- Product Scope for MVP
## In Scope
The MVP includes the following capabilities:
## Authentication
Users must be able to sign in securely using Firebase authentication. Only authenticated
users can upload documents, view their jobs, and download outputs.
PDF Upload
Authenticated users must be able to upload PDF files from the web application. The
system must validate file type and reject unsupported uploads.
## File Storage
Accepted source PDFs must be stored in the receiving storage path. Processed Excel
outputs must be stored in the processed storage path.
## Processing Trigger
An upload must create a processing job. The job must enter the backend workflow
automatically without requiring manual intervention.
## Extraction Pipeline
The system must process PDFs using a structured backend pipeline that includes:
- file retrieval
- document parsing and preprocessing
- extraction using Gemini

- schema mapping
- validation
- output generation
Document-Type Routing
The system must identify the expected extraction path for at least the initial supported
document categories, beginning with invoice extraction and a secondary research/report-
oriented mode.
## Job Status Tracking
Each upload must have a visible status, such as:
- uploaded
- queued
- processing
- completed
- failed
## Result Storage
Structured extraction results and job metadata must be persisted so the system can
display history, status, and download links.
## Excel Output Generation
A successful job must generate an Excel file and store it in processed storage.
## Download Experience
Users must be able to download the processed output from the web app once the job is
complete.
## Basic Failure Handling
Failed jobs must show failure state and a meaningful reason at the internal admin/operator
level.

## Internal Admin Visibility
Internal operators must have a limited management view for jobs and processing health.

## 10. Functional Requirements
FR-1 Authentication
The system shall require authentication before allowing document upload or history
access.
FR-2 Upload Acceptance
The system shall allow authenticated users to upload PDF files from the web application.
FR-3 Input Validation
The system shall validate upload type and reject unsupported formats.
FR-4 Source File Persistence
The system shall store source PDFs in receiving storage with metadata linking the file to a
job and user.
FR-5 Job Creation
The system shall create a processing job for every accepted upload.
FR-6 Queue-Based Processing
The system shall support asynchronous job execution so uploads do not block the
frontend experience.
FR-7 Extraction
The system shall extract structured data from supported document types using Gemini-
based AI extraction.

FR-8 Structured Mapping
The system shall map extracted content to a defined schema for each supported
document type.
FR-9 Validation
The system shall validate required fields and basic consistency rules before output
generation.
FR-10 Output Generation
The system shall generate an Excel file for successful jobs.
FR-11 Output Storage
The system shall store generated outputs in processed storage and associate them with
the original job.
FR-12 History View
The system shall provide users with a list of their uploaded files and processing states.
FR-13 Job Detail View
The system shall provide a detailed view showing file name, upload time, job status, output
availability, and basic extraction summary.
FR-14 Download
The system shall allow users to download completed Excel outputs.
FR-15 Failure State
The system shall mark jobs as failed if processing cannot complete and preserve an error
message for operators.

FR-16 Retry Support
The internal operator flow shall support retrying failed jobs.
FR-17 Basic Admin Observability
The internal admin/operator view shall expose queue and job-level health sufficient for
MVP troubleshooting.

## 11. User Experience Requirements
The product experience must feel fast, understandable, and trustworthy.
Users should never feel uncertain about what the system is doing. A successful UX for
PDFextract is one where:
- upload is obvious
- status is visible
- the system communicates progress
- the result is easy to retrieve
- failures do not feel silent or mysterious
The app should present itself as a professional document operations tool rather than a
generic AI playground. That means:
- restrained, credible visual design
- strong hierarchy
- clear status language
- visible artifacts and timestamps
- confidence-oriented wording
- no gimmicky AI messaging
The frontend should prioritize:
- dashboard clarity
- upload simplicity
- status legibility

- result discoverability
- admin troubleshooting usability
Detailed page-by-page behavior belongs in APP_FLOW.md, while design rules belong in
FRONTEND_GUIDELINES.md.

- Data and Output Requirements
The MVP must produce output that downstream users can act on.
For invoice documents, Excel output should preserve:
- top-level invoice metadata
- normalized financial fields
- line items in tabular structure
- traceability to source file and processing timestamp
For research/report-style documents, Excel output should preserve:
- document metadata
- sectioned extracted fields
- structured tables where available
- generated summary tabs if configured
At a minimum, every completed job must have:
- source file reference
- job ID
- user ID
- document type
- processing timestamps
- extraction status
- structured JSON result internally
- Excel artifact for user download
The system must store enough metadata to reconstruct what happened to a job without
manually searching logs first.


## 13. Success Metrics
The MVP should be considered successful if it meets the following business and
operational outcomes:
## Product Success
- Authenticated user can upload a PDF successfully
- Job is created and processed without manual backend intervention
- User can see status changes in the app
- Successful jobs produce downloadable Excel outputs
- Failure cases are visible rather than silent
## Demo Success
- A client-facing demo can show upload → processing → completed result cleanly
- At least invoice extraction works reliably enough for repeated demo runs
- The system appears designed, intentional, and production-minded
## Operational Success
- Internal team can inspect jobs and understand pipeline state
- Failed jobs can be identified and retried
- Storage paths and output artifacts remain organized and traceable
## Quality Success
- Extracted outputs are structurally correct for supported document types
- Required fields are validated before result publication
- Users do not receive malformed Excel files or broken downloads
The exact numerical KPIs, SLAs, and evaluation thresholds should be defined in
IMPLEMENTATION_PLAN.md and test plans once sample documents are finalized.


- Risks and Constraints
## Risk 1: Extraction Accuracy Variability
PDF layout differences can create inconsistent extraction quality. This is especially true for
non-standard invoices and research PDFs with dense or irregular structure.
Risk 2: Overbuilding the MVP
There is a strong temptation to add editing, schema builders, analytics, or multi-format
support too early. This would delay demo readiness.
## Risk 3: Silent Async Failure
Because the system is asynchronous, bad queue, trigger, or storage wiring can create
invisible failures unless status and observability are designed intentionally.
## Risk 4: Ambiguous Supported Types
If supported document types are not explicitly defined, AI tooling and developers may
invent extraction behavior or over-generalize the schema.
Risk 5: UI Drift
If the frontend is not constrained by design guidelines and app flow docs, it may turn into a
generic SaaS shell instead of a focused extraction workflow.
The project must therefore stay disciplined: clear scope, clear schema, clear status
model, and clear output path.

## 15. Assumptions
This PRD assumes:
- authenticated users are known and managed through Firebase Auth
- the primary upload format in v1 is PDF
- object storage paths are separated into receiving and processed areas

- async processing is mandatory
- invoice extraction is the most important first-class supported flow
- research/report extraction is secondary but planned within the same architecture
- users value downloadable Excel more than raw JSON in the MVP
- the internal team wants code-first architecture, not no-code assembly
- future extensibility matters, but present demo readiness matters more
These assumptions may be revised only through documented change control.

- Release Definition of Done
The MVP is done when all of the following are true:
- A user can authenticate.
- A user can upload a valid PDF.
- The system stores the file and creates a job.
- The job moves through async processing states.
- Gemini-based extraction runs for at least the initial supported schemas.
- The system validates the extraction result.
- A valid Excel file is generated for successful jobs.
- The processed output is stored and downloadable.
- Users can view upload history and job status.
- Internal operators can inspect failures and retry jobs.
- The end-to-end experience is stable enough for a repeated demo.
Anything less than that is not “done”; it is partial infrastructure.

## 18. Final Product Statement
PDFextract MVP is a focused AI document processing application for authenticated users
who need structured data extracted from PDFs. It is not a general AI sandbox and not a
broad enterprise content suite. Its purpose is specific: ingest PDFs, process them
asynchronously through an AI extraction pipeline, validate structured outputs, generate
downloadable Excel files, and provide status visibility throughout the workflow.
That focus is the product advantage. The narrower the product definition, the less room
there is for AI tooling or engineering work to drift into invented features, invented flows, or
invented architecture. This PRD exists to stop that drift before coding starts.
